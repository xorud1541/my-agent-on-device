# 사용성 검토 — "뭐부터 해야 하지?" 문제 (Capability Discoverability)

> 작성: 2026-06-13 / 브랜치: `feat/capability-discoverability`
> **베이스: `feat/screenshot-vision` (활성 개발선, main+24커밋 — 비전·영역캡처·macOS 포팅 포함).**
> ⚠️ 능력/한계의 진실은 **브랜치/런타임 의존**이다. main에는 비전·영역캡처가 없지만 활성선에는 있다.
> 이 사실 자체가 "단일 능력 원천(CapabilityManifest)을 런타임 탐지에 묶어야 한다"는 결론의 근거다.
> 목적: 사용자가 워크스페이스에서 **무엇을 할 수 있는지 / 무엇은 안 되는지**를
> 스스로 알 수 있게 만드는 개선안 검토 (구현 전 설계 리뷰)

## 1. 제보된 문제

> "한번 우리 에이전트 써봐" → 사용자: **"뭐부터 해야 하지?"**

빈 화면 앞에서 사용자가 행위의 진입점을 못 찾는다. 두 방향의 결핍이다.

1. **능력의 가시성 부족** — 이 에이전트가 *뭘 잘하고, 워크스페이스에 뭘 할 수 있는지*가 안 보인다.
2. **한계의 가시성 부족** — *뭘 못 하고 뭘 제공 못 하는지*를 (우리도, 사용자도) 명시적으로 인지하지 못한다.
   → 사용자는 안 되는 걸 시켜보고 실패로 경계를 배운다. (학습 비용 = 실패)

## 2. 현재 디스커버빌리티 표면 (코드 기준 사실)

| 위치 | 내용 | 한계 |
|---|---|---|
| `src/App.tsx` 빈 화면 | 워드마크 + 한 줄 소개 + **정적 제안 칩 4개** | 첫 메시지 후 **영구히 사라짐**. 워크스페이스 실제 내용과 무관 |
| `App.tsx` `SUGGESTIONS` | "다운로드 PDF 찾기 / 화면 캡처 / 배경 제거 / 이미지→PDF" | 워크스페이스가 비어있거나 사진이 없으면 → **막다른 길** (실패가 첫 경험) |
| `Composer.tsx` placeholder | "무엇을 도와드릴까요?" | 능력 힌트 없음 |
| `agent.rs` system_prompt 규칙 11 | "능력 질문엔 도구 목록을 나열" | **올바른 질문을 해야만** 도달. 한계(못 하는 일) 언급 없음 |
| 헤더 워크스페이스 칩 | 현재 폴더명 표시 | "왜 중요한지/이게 뭔지" 멘탈 모델 전달 안 됨 |

핵심: **능력 정보가 "물어봐야 나오는" 대화 경로에만 있고, 항상 보이는 영구 표면이 없다.**
그리고 **한계를 알려주는 표면은 아예 없다.**

## 3. 실제 능력 / 한계 (도구 레지스트리 기준, `tools/mod.rs`)

### ✅ 할 수 있는 일 (등록된 도구 17종) — **정확한 표현이 핵심**
능력은 "가능"만이 아니라 **조건(설정/권한/상태)**까지 함께 표기해야 정직하다 (2차 검증).

| 능력 | 정확한 범위 (과장 금지) | 조건/상태 |
|---|---|---|
| 파일 정리 | 목록/읽기/쓰기/이동/이름변경/복사/삭제(휴지통 경유), 검색 | 쓰기/삭제는 **워크스페이스 안에서만**, 읽기/검색은 전체 |
| 이미지 변환 | 리사이즈/포맷/회전 등 | — |
| `image_info` | **크기·포맷·파일크기 조회** (≠ "사진에 뭐가 있나" 판독) | — |
| 배경 제거 | ONNX 배경제거 | **`.ort` 모델 파일 필요** — 없으면 "배경제거 모델 없음" 에러(`image_ai.rs:166`) → "설정 필요" |
| PDF | **텍스트 레이어 추출** + 이미지 여러 장 → PDF 한 권 | 스캔본/OCR **불가**, 추출 길이 제한 |
| 화면 캡처 | **영역 드래그+크롭 → 첨부**(`bin/region_capture.rs`, 별도 프로세스) | macOS 화면 녹화 **권한** 필요 |
| **이미지 내용 설명/판독** | 첨부 이미지를 모델이 보고 설명 (멀티모달) | **조건부 — mmproj 페어링 시에만**(`vision_enabled=resolve_mmproj(..).is_some()`, `commands.rs:323`). 없으면 비활성 |
| 압축 | zip 생성/해제 | — |
| 워크스페이스 변경 | 대화로 작업 폴더 전환 | — |

> ⚠️ **베이스 정정 (활성선 기준):** 앞선 main 기준 검토에서 "비전·영역캡처는 한계"라 했으나,
> **활성 브랜치(`feat/screenshot-vision`)에선 둘 다 실제 능력**이다:
> - **비전(이미지 판독)**: mmproj가 페어링되면 동작하는 **조건부 능력**. mmproj 미존재 시에만 비활성 → 패널에서 "설정 필요/비활성"으로 표기.
> - **화면 캡처**: main의 전체 모니터가 아니라 **영역 드래그+크롭+첨부**(`region_capture.rs`).
> - 이 뒤집힘이 **CapabilityManifest를 런타임 탐지(resolve_mmproj/ort존재/OS)에 묶어야 하는 결정적 이유**다.
>   하드코딩하면 브랜치/구성에 따라 즉시 거짓이 된다.
> - 단, `image_info`는 여전히 비전이 아니다(크기·포맷 조회). "이 사진에 뭐가 있어?"는 위 비전 경로로 처리.

### ❌ 할 수 없는 일 (도구가 없음 = 능력의 경계)
- **인터넷/웹 검색/다운로드** — 로컬 온디바이스, 네트워크 도구 없음
- **이메일/메신저 발송**
- **워크스페이스 밖 쓰기/저장** — 쓰기·삭제는 워크스페이스 안으로 샌드박싱 (읽기/검색만 전체 허용)
- **문서 *내용* 편집** — Word/Excel/PPT, 또는 PDF 텍스트 편집·생성 불가 (PDF는 텍스트 추출 + 이미지→PDF만)
- **스캔 PDF/이미지 OCR** — `pdf_extract_text`는 텍스트 레이어만 (스캔본은 빈 결과)
- **다른 앱 자동화/클릭 제어, 시스템 설정 변경**
- **이미지 *생성*** (배경 제거·변환만, 생성 모델 없음), 오디오/영상 처리

> 이 ❌ 목록은 현재 우리도 코드 밖에 정리해둔 곳이 없다 → **문서화 자체가 이번 작업의 산출물.**

## 4. 개선 옵션 (브레인스토밍)

### 옵션 A — 정적 능력/한계 패널 (프론트엔드 only, 저위험)
- 빈 화면을 카테고리 그리드(파일/이미지/PDF/캡처)로 확장, 각 항목에 예시 프롬프트.
- 헤더에 **`?` "할 수 있는 일"** 영구 버튼 → 언제든 모달 재오픈.
- 모달 하단에 **"이건 아직 안 돼요"** 섹션(§3 ❌).
- 장점: React만, 모델 비용 0, 결정적, 즉시 가능. 단점: 정적 — 워크스페이스 실제 내용 미반영.

### 옵션 B — 워크스페이스 인지 동적 제안 (난이도 상향: 백엔드 신규 커맨드)
- 빈 화면에서 워크스페이스 내용을 읽어 내용 기반 제안 생성:
  - 이미지 N장 있음 → "사진 N장 배경 제거 / PDF로 묶기"
  - PDF 있음 → "이 PDF 내용 추출/요약"
  - **비어 있음** → "워크스페이스를 바꾸거나, 화면을 캡처해 보세요"
- ⚠️ **정정**: "`list_dir` 직접 IPC 재사용"은 불가능. 프론트 노출 커맨드 12개(`lib.rs:46~59`)에
  파일 도구가 **하나도 없다** — 도구는 전부 `send_message`(에이전트 루프) 안에서만 호출된다.
  따라서 B는 **신규 Tauri 커맨드**(예: `list_workspace_dir`) + `workspace::ensure_in_workspace` 가드 +
  `config-changed` 연동(워크스페이스 전환 시 갱신, `useAgent.ts:44` 채널 재사용) + `types.ts` 동기화가 필요.
  난이도는 "중간"이 아니라 **백엔드 신규 작업**. A와 분리해 별도 게이트로 진행 권장.
- **더 싼 대안(B-lite, 신규 커맨드 0)**: 빈 화면에 **"둘러보기" 버튼** 하나 → 기존 `send()`로
  "이 폴더에 뭐가 있는지 보고 뭘 할 수 있는지 알려줘" 사전정의 메시지를 던진다. 모델이 `list_dir`→요약 수행.
  단점은 2B 레이턴시/품질이지만 IPC 신설 리스크가 0. **B를 신설하기 전 이 경로부터 시도할 가치 있음.**
- 장점: **막다른 제안 문제를 정면 해결**, 첫 화면이 현실에 grounded.

### 옵션 C — 에이전트 자기인지 강화 (모델 주도, 주의 필요)
- 범위 밖 요청 시 정직하게 거절 + **가장 가까운 가능한 대안** 제시.
- ⚠️ 리스크: 2B 모델은 system prompt 비대화 시 라우팅 정확도 저하 (README/CLAUDE.md 명시 경고).
- **정정/보완 — 프롬프트보다 도구 결과 단이 2B엔 더 잘 먹힌다.** 코드에 이미 검증된 패턴이 있다:
  "2B는 프롬프트 규칙보다 직전 도구 결과의 지시를 더 잘 따른다"(`agent.rs` `not_found_msg`/회복 로직 주석).
  → 범위 밖/빈 워크스페이스 신호는 **system_prompt에 욱여넣기보다 도구 결과·UI 단에서 유도**하는 편이
  2B에 더 안전하고 효과적. system_prompt 변경이 불가피하면 **1~2줄**로만, `bench/toolcall_test.mjs` 회귀 검증 필수.

### 옵션 D — 가이드 레시피/원클릭 워크플로 (고난이도)
- "정리 마법사" 류 다단계 큐레이션. → 지금은 **오버스코프**, 보류 권장.

## 5. 권고안 (조합)

**1순위 (저위험·고효과): A + B 하이브리드**
- 영구 재오픈 가능한 **"할 수 있는 일 / 할 수 없는 일" 패널** (능력 + 한계 한 곳에).
- 빈 화면 제안을 **워크스페이스 인지형**으로 — 첫 화면이 막다른 길로 끝나지 않게.

**2순위 (정직성 요건): C를 신중하게**
- system_prompt에 범위 밖 1~2줄 + 대안 제시 지시. **bench 회귀 검증 통과를 게이트로.**

**보류: D** (스코프 큼)

### 왜 이 순서인가
- 사용자의 요청 절반은 "한계 인지"다 → ❌ 목록 문서화 + UI 노출이 핵심 산출물.
- 프론트엔드 표면은 2B 라우팅에 영향 0 → 안전하게 큰 개선.
- 시스템 프롬프트 변경은 효과적이지만 2B에서 가장 깨지기 쉬운 부분 → 검증 게이트 뒤로.

### 2차 검증(코덱스) 핵심 보강
1. **실제 1순위는 A 단독 MVP** — "A+B 하이브리드 1순위"는 과하다. B는 별도. 단, A만으론 "에이전트 자기인지"
   목표가 미달 → **C-lite(한계 문장 + negative bench)는 A와 함께** 최소한으로 가져간다.
2. **단일 능력 원천(CapabilityManifest) 필수 — drift 방지.** 능력표를 프론트에 하드코딩하면 또 어긋난다
   (이번 비전 오분류가 그 증거). **UI·system_prompt·테스트가 한 소스를 공유**하게 만든다. 도구 등록과
   manifest의 정합성을 단위 테스트로 강제.
3. **조건부 능력 표기** — 패널은 "가능"만이 아니라 **가능 / 설정 필요(removeBG `.ort`) / 권한 필요(macOS 캡처) /
   서버 준비중**을 구분해야 한다. (§3 표 참고)
4. **B는 LLM 생성보다 결정적 규칙이 낫다.** B-lite("둘러보기" 버튼)도 **리스크 0이 아니다** — 첫 경험을 느리고
   비결정적인 2B 턴 + 툴콜 실패 + 대화 히스토리 오염에 맡긴다. → 더 나은 최소안: **read-only `workspace_summary`
   커맨드**(파일 타입 개수·대표 항목만 결정적 반환) → 템플릿 제안. (이게 옵션 B의 올바른 형태)
5. **"호출할 도구 자체가 없는" 요청(웹검색/이메일/앱제어/이미지생성)은 도구 결과 단에서 안 잡힌다** —
   이쪽은 system_prompt 또는 UI/전처리 정책이 필요. C를 너무 뒤로 밀지 말고 **negative-case bench를 먼저 정의**하고
   그 범위 안에서만 프롬프트에 넣는다.
6. **워크스페이스 멘탈 모델을 명시** — "읽기/검색=전체 허용, 쓰기/삭제=워크스페이스 제한"의 비대칭을 안 쓰면
   프라이버시 기대와 실제가 어긋난다. 기본 워크스페이스가 홈 디렉터리면 동적 제안이 너무 넓고 민감 →
   **첫 진입은 "먼저 작업 폴더를 고르세요"**가 더 안전할 수 있다.
7. **제안 칩은 자동 실행(`App.tsx:106` `send(s)`)보다 composer에 채우기** — 사용자가 실행 전 문장을 보게.
8. **`?` 모달은 긴 기능표보다 3단 구성** — "파일 / 이미지·PDF / 로컬 제한".

## 6. 미해결/사용자 결정 필요
- 패널 진입 방식: 헤더 `?` 버튼 vs Composer `/help` vs 둘 다? (진단상 `?` 영구 버튼이 1순위, `/help`는 후순위로 못박는 게 일관적)
- 워크스페이스 첫 진입 시 **온보딩 오버레이**를 한 번 띄울까(첫 실행 한정)?
- 한계 목록의 톤: "안 돼요" 직설 vs "아직 안 돼요(로드맵 암시)"?
- **플랫폼 분기**: system_prompt 19행 "Windows PC", 경로 예시 `\Desktop` 백슬래시(26행)가 하드코딩.
  macOS 포팅 중이므로 능력/한계를 UI·프롬프트에 노출할 때 **OS 분기**가 필요 (예시 경로·캡처 동작·문구).
